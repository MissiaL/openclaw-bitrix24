# OpenClaw Bitrix24

[English](README.md) | **Русский**

Channel-плагин, подключающий ИИ-агента OpenClaw к Битрикс24: пользователи общаются с агентом прямо в мессенджере Битрикс24. В паре с любым REST-скиллом Битрикс24 агент сможет также управлять CRM, задачами, календарём и диском портала.

## Возможности

Всё перечисленное проверено на живом портале Битрикс24:

- **Переписка в обе стороны** в личных и групповых чатах, с конвертацией Markdown ⇄ BB-code и автоматической нарезкой длинных ответов.
- **Входящие файлы** — вложения пользователя (включая файлы с облачного диска) скачиваются и передаются агенту как полноценные медиа; настоящее имя файла и MIME-тип восстанавливаются из заголовков ответа скачивания.
- **Исходящие файлы** — агент может сформировать файл и отправить его в чат (загрузка через `imbot.v2.File.upload`, с подписью).
- **Цитирование** — когда пользователь отвечает цитатой, содержимое цитируемого сообщения подтягивается из кэша канала в контекст агента (Битрикс присылает только id сообщения).
- **Групповые чаты** — бот отвечает по упоминанию (`@ИмяБота`); Битрикс сам доставляет чат-боту групповые сообщения только при упоминании, так что агент не спамит в общий чат. При добавлении в чат бот присылает приветствие.
- **Индикатор «печатает…»** на всё время хода агента (обновляется каждые 8 секунд) и гаснет сразу после ответа.
- **Команды управления** — `/status`, `/new`, `/stop`, `/restart` с нативным слэш-меню в интерфейсе чата и allowlist-ом `commandUsers`.
- **Цели message-tool** — инструмент `message` агента умеет отправлять по адресам `2172` (пользователь) и `chat15762` (групповой чат).

Плагин работает с актуальным API чат-ботов Битрикс24 — **imbot.v2 (Chatbots 2.0)**, а не с устаревшими v1-методами `imbot.*`. Все события (новое сообщение, команда, добавление в чат, удаление бота, …) приходят на **единый вебхук-эндпоинт** аккаунта `/webhook/bitrix24/<accountId>` и маршрутизируются по полю `event` (например, `ONIMBOTV2MESSAGEADD`). Подписками на события Битрикс24 управляет сам при регистрации/обновлении бота — ручных `event.bind`/`event.unbind` нет.

## Требования

- **openclaw >= 2026.4** для современного вебхук-маршрута (`api.registerHttpRoute`, монтируется под `/webhook/bitrix24/` с `auth: 'plugin'` — Битрикс24 не умеет передавать gateway-токен, поэтому маршрут отключает *транспортную* авторизацию gateway; о прикладной защите см. [Безопасность](#безопасность)).
- На старых хостах плагин откатывается на легаси-механизм `registerService.router` — на транспортном уровне он так же не авторизован.

## Быстрый старт

### 1. Установите плагин

```bash
openclaw plugins install @openclaw/bitrix24
```

### 2. Получите вебхук

В портале Битрикс24: **Разработчикам → Другое → Входящий вебхук**.
Включите права: `imbot`, `im`, `disk`.
Скопируйте URL вебхука (вида `https://your-portal.bitrix24.ru/rest/1/abc123def/`).

### 3. Задайте переменную окружения

```bash
export BITRIX24_WEBHOOK_URL="https://your-portal.bitrix24.ru/rest/1/abc123def/"
```

### 4. Запустите агента

```bash
openclaw start
```

Плагин автоматически зарегистрирует чат-бота в портале при старте. Откройте мессенджер, найдите бота («OpenClaw Agent») и напишите ему.

### 5. Проверьте

Выполните `/b24status` в OpenClaw:

```
/b24status
# Bitrix24 Accounts:
# - default (your-portal.bitrix24.ru): connected
```

## Конфигурация

Плагин поддерживает два способа авторизации: URL вебхука (просто) и OAuth (мультипортальность).

### Вариант А: URL вебхука (быстрая настройка)

Достаточно переменной окружения `BITRIX24_WEBHOOK_URL` — без правки конфига.

Либо в `openclaw.yaml`:

```yaml
channels:
  bitrix24:
    webhookUrl: "https://your-portal.bitrix24.ru/rest/1/abc123def/"
```

### Публичный URL (обработчики событий)

Битрикс24 обращается к боту по HTTP (единый `webhookUrl`, передаваемый в `imbot.v2.Bot.register`/`imbot.v2.Bot.update`), поэтому плагину нужен внешне доступный базовый URL gateway. Порядок разрешения:

1. `channels.bitrix24.publicUrl` (конфиг)
2. Переменная окружения `BITRIX24_PUBLIC_URL`
3. `gateway.externalUrl` (легаси; удалён из схемы конфига в openclaw 2026.6, оставлен как фолбэк)
4. `http://localhost:18789` (локальный дефолт)

Задать:

```bash
openclaw config set channels.bitrix24.publicUrl https://bot.example.com
```

Если `publicUrl` изменился после регистрации бота, плагин заметит это на следующем старте и вызовет `imbot.v2.Bot.update` с новым `webhookUrl` — Битрикс24 сам перенацелит все подписки `ONIMBOTV2*` на новую базу. Последняя зарегистрированная база хранится в `channels.bitrix24.registeredWebhookBase.<accountId>`; ключ управляется плагином, руками не редактировать.

В imbot.v2 существует и режим `fetch` (поллинг `imbot.v2.Event.get`) — этот плагин всегда регистрируется в режиме `webhook`; `fetch` возможен как опция в будущем для деплоев без публичного URL.

### Безопасность

Эндпоинт `/webhook/bitrix24/...` публично доступен на уровне gateway: он зарегистрирован с `auth: 'plugin'` (и легаси-фолбэк ведёт себя так же), т.е. gateway-токен не проверяется — Битрикс24 не умеет его передавать.

Этот пробел плагин закрывает сам — **TOFU-аутентификацией событий** (trust-on-first-use). Каждое входящее событие несёт top-level `auth.application_token`, выданный порталом. При первом реальном событии аккаунта плагин запоминает и durable-сохраняет токен (`channels.bitrix24.accounts[].applicationToken`, управляется плагином). Дальше каждое событие сверяется с запиненным токеном; несовпадение или отсутствие токена — HTTP 403 и отбрасывание до агента.

Это защищает от вброса поддельных событий: зная только `publicUrl`, атакующий не подделает `application_token` портала.

**Оговорка:** TOFU защищает только *после* того, как первое настоящее событие принято. Если атакующий успеет прислать поддельное первое событие раньше портала, будет запинен его токен, а настоящие события начнут отклоняться. Митигируется подключением портала (первым реальным событием) в доверенном окружении до широкой публикации `publicUrl`.

Помимо TOFU: содержимое входящих сообщений — недоверенный ввод; держите `publicUrl` неугадываемым (путь за реверс-прокси или allow-list IP-диапазонов Битрикс24) как защиту в глубину.

### Вариант Б: Мультиаккаунт / OAuth

```yaml
channels:
  bitrix24:
    accounts:
      - id: main
        webhookUrl: "https://portal-a.bitrix24.ru/rest/1/secret1/"
        bot:
          name: "Sales Bot"
          color: AZURE
          workPosition: "Sales Assistant"
          # bot.clientId выводится автоматически из секрета вебхука

      - id: support
        domain: portal-b.bitrix24.ru
        accessToken: "your-oauth-access-token"
        refreshToken: "your-oauth-refresh-token"
        clientId: "app.xxxxxxxx.xxxxxxxx"      # clientId OAuth-приложения
        clientSecret: "your-client-secret"     # clientSecret OAuth-приложения
        bot:
          name: "Support Bot"
          color: GREEN
          clientId: "stable-secret-bot-client-id"
        dmPolicy: paired          # "open" (по умолчанию) или "paired"
        textChunkLimit: 3000      # макс. символов в одном сообщении (по умолчанию 18000)
```

**Порядок разрешения авторизации** (для аккаунта default):
1. `webhookUrl` или `accessToken` аккаунта
2. Глобальный `channels.bitrix24.webhookUrl`
3. Переменная окружения `BITRIX24_WEBHOOK_URL`

Остальные аккаунты используют только собственные реквизиты.

## Токен бота (`botToken`)

Методы imbot.v2 требуют стабильный секретный `botToken`, привязанный к создателю бота (в v1 эту роль играл `CLIENT_ID`; поле конфига по-прежнему называется `bot.clientId` для совместимости).

- для вебхук-аккаунтов выводится автоматически: `md5(нормализованный webhookUrl)`
- для OAuth-аккаунтов задаётся явно в `accounts[].bot.clientId`
- одно и то же значение передаётся как `botToken` в `imbot.v2.Bot.register/update/unregister`, `imbot.v2.Chat.Message.send/update/delete`, `imbot.v2.Chat.InputAction.notify`, `imbot.v2.File.upload/download`, `imbot.v2.Command.register/list`

Не публикуйте это значение: это часть контура управления ботом.

## Команды в чате

Плагин регистрирует на портале нативное слэш-меню (идемпотентно, при каждом старте аккаунта) через `imbot.v2.Command.register`:

| Команда | Действие |
|---|---|
| `/status` | Статус агента и лимиты подписки |
| `/new` | Начать новую сессию |
| `/stop` | Прервать текущую задачу |
| `/restart` | Перезапустить gateway OpenClaw |

Сами команды — встроенные в OpenClaw; канал решает лишь, *кому* они доступны, через `commandUsers`:

```yaml
channels:
  bitrix24:
    accounts:
      - id: default
        commandUsers: ["2172"]   # id пользователей Битрикс; "*" — всем
```

По умолчанию (`commandUsers` пуст) команды **выключены**: бот доступен всем сотрудникам портала, и открытый `/restart` был бы дырой. Для неавторизованных пользователей `/команда` — обычный текст для агента.

Вызовы из меню приходят событием `ONIMBOTV2COMMANDADD`; набранная вручную `/команда` может продублироваться обычным событием сообщения — роутер вебхука дедуплицирует по id сообщения, агент видит каждое сообщение ровно один раз.

## Архитектура

```
+-------------------+         +---------------------------+
|   Битрикс24       |         |   Агент OpenClaw          |
|   Мессенджер      |         |                           |
|                   |         |  +---------+  +---------+ |
|  Сообщение юзера  | ------> |  | Webhook |->| Агент   | |
|                   |  POST   |  | Server  |  | (LLM)   | |
|                   |         |  +---------+  +----+----+ |
|  Ответ бота       | <------ |                    |      |
|                   | imbot.v2|  +---------+       |      |
|                   | .Chat.  |  | Bitrix24|<------+      |
|                   | Message.|  | Client  |  REST API    |
|                   | send    |  +---------+  (скилл)     |
+-------------------+         +---------------------------+
```

**Путь сообщения:**

1. Пользователь пишет боту в мессенджере Битрикс24
2. Битрикс24 шлёт `ONIMBOTV2MESSAGEADD` на единый вебхук-эндпоинт (`/webhook/bitrix24/<accountId>`), событие проходит TOFU-проверку `application_token`
3. Вебхук-сервер разбирает событие, конвертирует BB-code в Markdown; вложения (`params.FILE_ID`) скачиваются и передаются агенту как локальные медиа; цитата (`params.REPLY_ID`) резолвится из кэша недавних сообщений в контекст агента
4. Сообщение уходит агенту OpenClaw; индикатор «печатает…» (`imbot.v2.Chat.InputAction.notify`) поддерживается весь ход
5. Агент формирует ответ (при необходимости вызывая REST API Битрикс24 через скилл)
6. Ответ конвертируется из Markdown в BB-code и при необходимости нарезается
7. Бот отвечает через `imbot.v2.Chat.Message.send`; сгенерированные агентом файлы — одним вызовом `imbot.v2.File.upload`

## Разработка

### Требования

- Node.js >= 20 (см. `engines` в `extensions/bitrix24/package.json`)
- npm

### Установка

```bash
git clone https://github.com/rsvbitrix/openclaw-bitrix24.git
cd openclaw-bitrix24
npm install
```

### Запуск из рабочей копии

Два способа без публикации в npm:

- **Dev (TypeScript-исходники напрямую):** добавьте каталог плагина в `plugins.load.paths`:

  ```yaml
  plugins:
    load:
      paths:
        - /path/to/openclaw-bitrix24/extensions/bitrix24
  ```

  OpenClaw возьмёт точку входа из `openclaw.extensions` в `extensions/bitrix24/package.json` (`./src/index.ts`) и выполнит TypeScript без сборки. Нужен openclaw >= 2026.4.

- **Prod (собранный билд, как опубликованный пакет):**

  ```bash
  cd extensions/bitrix24
  npm install
  npm run build
  openclaw plugins install .
  ```

  Собирается `dist/`, плагин ставится из скомпилированной точки входа `openclaw.runtimeExtensions` (`./dist/extensions/bitrix24/src/index.js`).

### Сборка / тесты / линт

```bash
npm run build
npm test              # один прогон
npm run test:watch    # watch-режим
npm run lint
```

## Права (scopes)

Вебхуку или OAuth-приложению Битрикс24 нужны:

| Scope | Обязателен | Для чего |
|---|---|---|
| `imbot` | Да | Регистрация бота, отправка сообщений от его имени |
| `im` | Да | Сообщения, чаты, файлы в чатах |
| `disk` | Да | Загрузка/скачивание файлов |
| `crm` | Для скилла | CRM (сделки, контакты, лиды, компании) |
| `task` | Для скилла | Задачи |
| `calendar` | Для скилла | Календарь |
| `user` | Для скилла | Поиск пользователей, отделы |
| `department` | Для скилла | Оргструктура |

**Минимум для канала:** `imbot`, `im`, `disk`.

## Диагностика проблем

### Бот не появляется в мессенджере

- Проверьте, что `BITRIX24_WEBHOOK_URL` задан и валиден.
- Убедитесь, что у вебхука включён scope `imbot`.
- Выполните `/b24status` — при статусе «connected» бот должен быть в списке контактов в разделе «Боты и приложения».
- Посмотрите логи агента на ошибки регистрации.

### Сообщения не доходят

- Агент должен быть доступен из интернета: Битрикс24 шлёт все события `ONIMBOTV2*` на `{publicUrl}/webhook/bitrix24/{accountId}`.
- Проверьте, что `channels.bitrix24.publicUrl` (или `BITRIX24_PUBLIC_URL`) — публично доступный HTTPS-URL.
- Убедитесь, что вебхук в Битрикс24 не отозван.
- Если аккаунт раньше подключался с другого `publicUrl` и первое событие пришло не от настоящего портала, TOFU мог запинить чужой `application_token` и отклонять настоящие события с HTTP 403 — см. [Безопасность](#безопасность). Очистите `accounts[].applicationToken`, следующее реальное событие перепинит токен.
- Для трассировки включите `BITRIX24_DEBUG=1` в окружении gateway: логируется сырое тело события, разобранное сообщение, диспетчеризация агенту и доставка ответа. Содержит переписку пользователей (PII) — включайте только на время отладки.

### Ошибки лимитов (`QUERY_LIMIT_EXCEEDED` / HTTP 503 / HTTP 429)

Документация Битрикс24 расходится в сигнале (503 + `QUERY_LIMIT_EXCEEDED` против 429), поэтому клиент ретраит с экспоненциальным бэкоффом оба варианта, плюс `OVERLOAD_LIMIT`/`OPERATION_TIME_LIMIT`. По умолчанию действует token-bucket лимитер 2 req/s.

### Длинные сообщения обрезаются

Плагин автоматически режет сообщения по границам абзацев/предложений. Настройте `textChunkLimit` (по умолчанию 18000).

### Не загружаются файлы

- Проверьте scope `disk` (imbot.v2 всё ещё требует его, хотя `imbot.v2.File.upload` — один вызов без ручного управления папками Диска).
- Файлы больше 100 МБ падают с `FILE_TOO_LARGE`.
- **Входящие файлы:** проверено вживую — вложение приходит как `message.params.FILE_ID: ["<driveFileId>"]` (без имени и размера; настоящие имя и MIME-тип берутся из заголовков ответа скачивания). Формы `params.files` из документации оставлены как фолбэки. Если вложение не распознаётся — включите `BITRIX24_DEBUG=1` и приложите к issue обезличенное сырое событие.

### Ответы бота с «кракозябрами» разметки

Плагин конвертирует Markdown в BB-code (родной формат Битрикс24). Если видите сырые BB-теги — заведите issue с текстом входного сообщения.

### Истёк OAuth-токен

Для OAuth-аккаунтов задавайте и `accessToken`, и `refreshToken`. Автообновление требует настроенных `clientId` и `clientSecret`.
