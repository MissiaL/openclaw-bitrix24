# Автоматические персональные агенты Bitrix24

**Дата:** 22 июля 2026 года

**Статус:** согласовано к реализации в узком scope плагина

**Ветка:** `codex/bitrix-dynamic-agents`

## Задача

Адаптировать механизм `dynamicAgentCreation` из bundled-плагина Feishu для внешнего Bitrix24-плагина. В интерфейсе Bitrix24 остаётся один бот на каждый настроенный аккаунт, но для каждого сотрудника в личном диалоге автоматически создаётся отдельный внутренний агент с собственными workspace, сессиями и долговременной памятью.

Единицей изоляции является пара **«аккаунт Bitrix24-бота × сотрудник»**. Поэтому память сотрудника не смешивается ни с памятью другого сотрудника, ни с памятью того же сотрудника в другом функциональном боте.

Эта доработка является общей возможностью плагина. Включение функции и настройка конкретных аккаунтов на сервере заказчика выполняются отдельно и в текущую работу не входят.

## Пользовательская модель

- Для каждого аккаунта Bitrix24 остаётся один видимый бот.
- В личном диалоге первое сообщение создаёт персональный внутренний агент и сразу же обрабатывается им.
- Последующие сообщения сотрудника тому же боту попадают в тот же агент.
- У каждого персонального агента отдельны `workspace`, `agentDir`, история сессий и долговременная память.
- Групповые чаты продолжают обслуживаться существующим базовым агентом и не создают персональных агентов.
- Новый сотрудник не требует ручной привязки: она создаётся лениво при первом личном сообщении.

## Почему одной `session.dmScope` недостаточно

`session.dmScope: "per-channel-peer"` разделяет историю личных диалогов по пользователям внутри канала, но не разделяет файлы рабочего пространства агента: `MEMORY.md`, `USER.md`, каталог `memory/` и результаты работы.

Персональный `agentId` даёт отдельные workspace, память и служебное состояние. Для точной peer-привязки дополнительно задаётся binding-level `session.dmScope: "per-account-channel-peer"`; это второй защитный слой, а не замена отдельному агенту.

## Основа решения: паттерн Feishu

Решение адаптирует механизм `dynamicAgentCreation` из bundled-плагина Feishu:

- детерминированный идентификатор агента;
- запись агента и точной peer-привязки одной атомарной мутацией конфигурации;
- повторная проверка правил и лимита внутри блокировки мутации;
- идемпотентность при одновременных первых сообщениях;
- повторное разрешение маршрута после записи, чтобы первое сообщение не ушло базовому агенту;
- отдельные шаблоны путей для `workspace` и `agentDir`;
- защитный лимит числа динамических агентов.

Код Feishu нельзя переносить буквально: внешний Bitrix24-плагин не импортирует `openclaw/*`. Все возможности OpenClaw используются через внедрённый `api.runtime`, как и в остальном коде плагина.

В Bitrix24 аккаунт уже может быть привязан к функциональному базовому агенту. Поэтому создание выполняется только когда:

1. для сотрудника ещё нет точной direct-peer-привязки;
2. текущий маршрут ведёт в настроенный `sourceAgentId` аккаунта;
3. динамическое создание и запись конфигурации разрешены;
4. `dmPolicy` аккаунта равен `open`.

## Конфигурация

В конфигурацию канала и каждого элемента `accounts[]` добавляются:

```json
{
  "configWrites": true,
  "dynamicAgentCreation": {
    "enabled": true,
    "sourceAgentId": "b24-example",
    "workspaceTemplate": "~/.openclaw/ws-b24/users/{accountId}/{userId}",
    "agentDirTemplate": "~/.openclaw/agents/{agentId}/agent",
    "bootstrapFiles": [
      "AGENTS.md",
      "SOUL.md",
      "TOOLS.md",
      "IDENTITY.md",
      "HEARTBEAT.md"
    ],
    "maxAgents": 500
  }
}
```

Правила конфигурации:

- Значение аккаунта заменяет одноимённое значение уровня канала; объект `dynamicAgentCreation` не сливается частично.
- По умолчанию `configWrites` и `dynamicAgentCreation.enabled` выключены, поэтому существующие установки не меняют поведение после обновления.
- `sourceAgentId` обязателен при `enabled: true` и должен существовать в `agents.list`.
- Поддерживаются шаблоны `{accountId}`, `{userId}` и `{agentId}`.
- `bootstrapFiles` содержит только относительные пути без `..`; символические ссылки и выход за пределы workspace базового агента отклоняются.
- `maxAgents` считается отдельно для каждого аккаунта Bitrix24.
- Если функция включена, но безопасное создание невозможно, сообщение не отправляется базовому агенту: это предотвращает повторное смешивание памяти.

Обе копии JSON Schema в `openclaw.plugin.json` обновляются синхронно: верхнеуровневая `configSchema` и `channelConfigs.bitrix24.schema`.

## Идентичность и маршрутизация

Каноническим direct-peer ID является `fromUserId`, а не `dialogId`. `dialogId` остаётся адресом доставки ответа. Для групп peer ID остаётся `dialogId` вида `chatNNN`.

Идентификатор персонального агента вычисляется детерминированно из `accountId + NUL + fromUserId`. Он содержит безопасный короткий префикс аккаунта и усечённый SHA-256, например `bitrix24-tkp-a1b2c3d4`. Одинаковый пользователь в разных аккаунтах получает разные агенты.

Точная привязка имеет вид:

```json
{
  "agentId": "bitrix24-tkp-a1b2c3d4",
  "match": {
    "channel": "bitrix24",
    "accountId": "tkp",
    "peer": { "kind": "direct", "id": "403" }
  },
  "session": { "dmScope": "per-account-channel-peer" }
}
```

## Обработка первого сообщения

1. Плагин принимает уже аутентифицированное событие Bitrix24 и определяет аккаунт, тип чата и `fromUserId`.
2. Групповое сообщение сразу проходит по существующему маршруту.
3. Для личного сообщения плагин читает актуальную runtime-конфигурацию.
4. Если точная peer-привязка уже существует, создание не выполняется.
5. Если маршрут не ведёт в `sourceAgentId`, ручная более точная маршрутизация сохраняет приоритет.
6. Если функция включена, но `dmPolicy` не `open`, config writes недоступны, конфигурация невалидна или достигнут лимит, обработка завершается безопасным отказом без shared fallback.
7. Внутри `runtime.config.mutateConfigFile({ base: "runtime", afterWrite: { mode: "auto" } })` повторно проверяются флаги, policy, маршрут, существование привязки и лимит.
8. Создаются workspace и `agentDir`, копируются только разрешённые bootstrap-файлы, затем одной мутацией добавляются `agents.list[]` и точная `bindings[]`.
9. После записи плагин перечитывает runtime-конфигурацию, заново разрешает маршрут и обрабатывает первое сообщение новым агентом.

При гонке нескольких сообщений только первая мутация создаёт агент; остальные видят готовую привязку. Если детерминированный агент уже существует, но binding отсутствует, восстанавливается только binding. Неоднозначные или вручную созданные агенты автоматически не перепривязываются и не удаляются.

## Инициализация workspace

В персональный workspace копируются только явно перечисленные `bootstrapFiles` из workspace базового агента. Не копируются:

- `USER.md` и `MEMORY.md` базового агента;
- каталог `memory/`, сессии и служебное состояние;
- `out/`, входящие файлы и ранее созданные результаты.

Плагин создаёт новый `USER.md` с Bitrix ID, именем сотрудника и аккаунтом бота. Остальная персональная память возникает уже внутри его собственного workspace.

Эта работа не меняет содержимое bootstrap-файлов, скрипты агентов, генераторы документов или прикладные правила функциональных агентов.

## Ошибки и безопасность

- Ошибка создания workspace, копирования bootstrap-файлов или записи конфигурации означает безопасный отказ без fallback в общий агент.
- Пользователю отправляется короткое нейтральное сообщение без внутренних путей и технических подробностей.
- Полная причина записывается структурированно с `accountId`, `userId` и стадией; текст сообщения и имя сотрудника в журнал не попадают.
- Выключение функции не удаляет уже созданные агенты и bindings.
- Архивация или удаление персонального агента остаётся отдельной административной операцией.

## Локальная проверка

Обязательные сценарии:

1. функция полностью выключена;
2. функция включена, но config writes выключены;
3. групповое сообщение;
4. policy не `open`;
5. уже существующая точная привязка;
6. ручной маршрут не в `sourceAgentId`;
7. успешное создание агента и binding;
8. первое сообщение после создания маршрутизируется новому агенту;
9. повторное и одновременное первое сообщение не создают дубликаты;
10. runtime-конфигурация новее входного снимка;
11. достижение per-account `maxAgents` и отсутствие shared fallback;
12. одинаковый `userId` в разных аккаунтах создаёт разные агенты;
13. существующий детерминированный агент без binding получает только недостающую привязку;
14. запрещённый bootstrap-путь и отсутствие копирования памяти/результатов;
15. direct routing использует `fromUserId`, а доставка — `dialogId`;
16. обе копии manifest schema совпадают;
17. в исходниках и сборке отсутствуют импорты `openclaw/*`.

Приёмка ограничена локальными командами `npx vitest run`, `npm run lint`, `npm run build`, `git diff --check` и проверкой импортов.

## Границы этой доработки

Входит в работу:

- общий механизм динамических персональных агентов в Bitrix24-плагине;
- конфигурационная схема функции на уровне канала и аккаунта;
- отдельные workspace, сессии и память;
- безопасный bootstrap из соответствующего базового агента;
- локальные unit/integration-тесты и сборка.

Не входит в работу:

- подключение к серверу заказчика, изменение его конфигурации и live-проверка;
- включение функции для конкретных клиентских аккаунтов;
- изменение скриптов агентов и генераторов документов;
- изменение НДС, шаблонов DOCX/PDF и прикладных правил ТКП;
- отдельный видимый бот для каждого сотрудника;
- общая память одного сотрудника между разными функциональными ботами;
- автоматическое удаление данных уволенного сотрудника;
- веб-интерфейс управления персональными агентами.
