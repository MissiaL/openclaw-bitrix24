# Автоматические персональные агенты Bitrix24

**Дата:** 22 июля 2026 года

**Статус:** проект решения для согласования

**Ветка:** `codex/bitrix-dynamic-agents`

## Задача

Сохранить по одному видимому Bitrix24-боту на каждую корпоративную функцию, но дать каждому сотруднику отдельную память и рабочее пространство. Новый сотрудник не должен требовать ручной настройки: персональный агент создаётся при его первом личном сообщении соответствующему боту.

Для текущего клиента схема применяется ко всем четырём функциональным ботам:

| Аккаунт Bitrix24 | Базовый агент | Назначение |
| --- | --- | --- |
| `default` | `b24-general` | общий помощник |
| `suppliers` | `b24-suppliers` | поставщики |
| `lawyer` | `b24-lawyer` | юридические задачи |
| `tkp` | `b24-tkp` | ТКП |

Единицей изоляции является пара **«аккаунт бота × сотрудник»**. Поэтому память сотрудника в ТКП-боте не смешивается ни с памятью другого сотрудника, ни с его памятью в юридическом боте.

## Пользовательская модель

- В интерфейсе Bitrix24 остаются те же четыре бота. Для каждого сотрудника новые боты не регистрируются.
- В личном диалоге первое сообщение создаёт персональный внутренний агент и сразу же обрабатывается им.
- Последующие сообщения того же сотрудника тому же боту попадают в тот же агент.
- Личные предпочтения — стиль текста, подпись, ФИО и должность, порядок разделов и стандартные формулировки — записываются только в персональную память.
- Корпоративные правила, реквизиты, логика НДС и официальный шаблон не становятся пользовательскими настройками и не копируются в персональную память.
- Групповые чаты продолжают обслуживаться базовым функциональным агентом и не создают персональных агентов.

Если в компанию приходит новый сотрудник, достаточно дать ему доступ к существующему Bitrix24-боту. Привязка создастся лениво при первом личном сообщении.

## Почему одной `session.dmScope` недостаточно

Текущий `session.dmScope: "per-channel-peer"` уже разделяет историю личных диалогов по пользователям внутри канала. Это защищает краткосрочный контекст сессии, но не разделяет файлы рабочего пространства агента — в том числе `MEMORY.md`, `USER.md`, каталог `memory/` и результаты работы.

Персональный агент даёт отдельные `workspace`, `agentDir` и хранилище сессий. Для точной peer-привязки дополнительно задаётся binding-level `session.dmScope: "per-account-channel-peer"`; это второй защитный слой, а не замена отдельному агенту.

## Основа решения: паттерн Feishu

Решение адаптирует механизм `dynamicAgentCreation` из bundled-плагина Feishu:

- детерминированный идентификатор агента;
- запись агента и точной peer-привязки одной атомарной мутацией конфигурации;
- повторная проверка правил и лимита внутри блокировки мутации;
- идемпотентность при одновременных первых сообщениях;
- повторное разрешение маршрута после записи, чтобы первое сообщение не ушло базовому агенту;
- отдельные шаблоны путей для `workspace` и `agentDir`;
- защитный лимит числа динамических агентов.

Код Feishu нельзя переносить буквально: внешний Bitrix24-плагин не имеет права импортировать `openclaw/*`. Все возможности OpenClaw используются через внедрённый `api.runtime`, как и остальной код плагина.

Есть и важное отличие маршрутизации. Feishu создаёт агента только при маршруте `matchedBy: "default"`. У клиента Bitrix24 уже есть привязки аккаунтов к базовым агентам, поэтому Bitrix-реализация проверяет одновременно:

1. для сотрудника ещё нет точной direct-peer-привязки;
2. текущий маршрут ведёт в настроенный `sourceAgentId` данного аккаунта;
3. динамическое создание и запись конфигурации разрешены для этого аккаунта.

## Конфигурация

В конфигурацию канала и каждого элемента `accounts[]` добавляются:

```json
{
  "configWrites": true,
  "dynamicAgentCreation": {
    "enabled": true,
    "sourceAgentId": "b24-tkp",
    "workspaceTemplate": "~/.openclaw/ws-b24/users/{accountId}/{userId}",
    "agentDirTemplate": "~/.openclaw/agents/{agentId}/agent",
    "bootstrapFiles": [
      "AGENTS.md",
      "SOUL.md",
      "TOOLS.md",
      "IDENTITY.md",
      "HEARTBEAT.md"
    ],
    "maxAgents": 500
  }
}
```

Правила конфигурации:

- Значение аккаунта заменяет одноимённое значение уровня канала; для `dynamicAgentCreation` используется весь объект аккаунта без частичного слияния.
- По умолчанию `configWrites` и `dynamicAgentCreation.enabled` выключены. Существующие установки не меняют поведение после обновления плагина.
- Создание разрешено только когда оба флага включены.
- `sourceAgentId` обязателен при `enabled: true` и должен существовать в `agents.list`.
- `{accountId}`, `{userId}` и `{agentId}` поддерживаются в шаблонах путей.
- `bootstrapFiles` содержит только относительные пути без `..`; символические ссылки и выход за пределы workspace базового агента отклоняются.
- Лимит `maxAgents` считается отдельно для каждого аккаунта Bitrix24. Для клиента устанавливается 500 персональных агентов на функционального бота. При достижении лимита сообщение не отправляется общему агенту: пользователь получает нейтральное сообщение о временной недоступности, а в журнал пишется структурированная ошибка.

Обе копии JSON Schema в `openclaw.plugin.json` обновляются синхронно: верхнеуровневая `configSchema` и `channelConfigs.bitrix24.schema`.

## Идентичность и маршрутизация

Каноническим direct-peer ID является `fromUserId`, а не `dialogId`. `dialogId` остаётся адресом доставки ответа. Для групп каноническим peer ID остаётся `dialogId` вида `chatNNN`.

Идентификатор персонального агента вычисляется детерминированно из `accountId + NUL + fromUserId`. Он содержит безопасный короткий префикс аккаунта и усечённый SHA-256, например `bitrix24-tkp-a1b2…`. Это обеспечивает ограниченную длину, отсутствие недопустимых символов и уникальность между аккаунтами.

Точная привязка имеет вид:

```json
{
  "agentId": "bitrix24-tkp-a1b2c3d4",
  "match": {
    "channel": "bitrix24",
    "accountId": "tkp",
    "peer": { "kind": "direct", "id": "403" }
  },
  "session": { "dmScope": "per-account-channel-peer" }
}
```

## Обработка первого сообщения

1. Плагин принимает уже аутентифицированное событие Bitrix24 и определяет аккаунт, тип чата и `fromUserId`.
2. Групповые сообщения сразу проходят по существующему маршруту.
3. Для личного сообщения плагин читает актуальную runtime-конфигурацию, а не только снимок `api.config`.
4. Если точная peer-привязка уже существует, создание не выполняется.
5. Если маршрут не ведёт в `sourceAgentId`, плагин не перехватывает его: ручная более точная маршрутизация имеет приоритет.
6. Для `dmPolicy: "open"` доверенным источником считается сотрудник портала, пришедший через проверенный webhook соответствующего аккаунта. Если динамическое создание включено, но policy отличается, запрос завершается безопасным отказом и не передаётся базовому агенту.
7. Внутри `runtime.config.mutateConfigFile({ base: "runtime", afterWrite: { mode: "auto" } })` повторно проверяются флаги, policy, маршрут, существование привязки и лимит.
8. Создаются workspace и agentDir, копируются только разрешённые bootstrap-файлы, затем в одной мутации добавляются `agents.list[]` и точная `bindings[]`.
9. После записи плагин перечитывает runtime-конфигурацию, заново разрешает маршрут и обрабатывает это же первое сообщение новым агентом.

При гонке нескольких сообщений только первая мутация создаёт агент; остальные видят готовую привязку. Если агент уже существует, но binding отсутствует, безопасно восстанавливается только binding. Неоднозначные или вручную созданные агенты автоматически не перепривязываются и не удаляются.

## Рабочие пространства и корпоративные файлы

В персональный workspace попадают только корпоративные инструкции, необходимые для старта. Не копируются:

- `USER.md` и `MEMORY.md` базового агента;
- каталог `memory/`, `DREAMS.md`, сессии и служебное состояние;
- `out/` и ранее созданные документы;
- входящие файлы и `.openclaw/media/`.

Для сотрудника создаётся новый `USER.md` с именем, Bitrix ID и аккаунтом бота; персональные `MEMORY.md`, `memory/` и `out/` возникают уже внутри его workspace.

Корпоративные артефакты не копируются в пользовательские рабочие пространства. Для ТКП это прежде всего генератор и официальный DOCX-шаблон. Они остаются в стабильном общем каталоге, на который ссылаются bootstrap-инструкции персональных агентов. На сервере общий каталог и шаблон делаются доступными агенту только на чтение на уровне прав файловой системы. Благодаря этому сотрудники могут менять содержание конкретного предложения и свои предпочтения, но не корпоративный бланк.

Изменения корпоративных bootstrap-инструкций для уже созданных агентов выполняются отдельной явной командой синхронизации, которая перезаписывает только перечисленные `bootstrapFiles` и никогда не затрагивает персональные файлы. Автоматическая синхронизация на каждом сообщении не применяется.

## Правила ТКП и НДС

Налоговая логика закрепляется кодом генератора и тестами, а не только текстовой подсказкой модели:

- ставка НДС для клиента — 22%;
- если цена включает НДС, налоговая часть считается как `сумма × 22 / 122`;
- сумма без НДС равна `сумма с НДС − НДС`;
- округление денежных значений выполняется до копеек по одному правилу в генераторе;
- модель не создаёт DOCX произвольным `python-docx`-скриптом, а вызывает только проверенный генератор;
- корпоративный шаблон обязателен и не может быть заменён персональным файлом;
- перед отправкой выполняются структурные проверки DOCX/PDF и визуальная проверка отрендеренного документа.

Персональная память может хранить предпочтение формулировок или подписи, но не может переопределить ставку НДС, формулы, реквизиты и бланк.

## Ошибки и безопасность

- Ошибка создания workspace, копирования bootstrap-файлов или записи конфигурации означает отказ обработки этим личным агентом. Fallback в общий агент запрещён, потому что он снова смешал бы память сотрудников.
- Пользователю отправляется короткое сообщение без внутренних путей и технических подробностей; полная причина записывается в журнал с `accountId`, `userId` и стадией.
- Имена и текст сообщений не попадают в обычные структурные логи.
- Удаление сотрудника из Bitrix24 прекращает его новые входящие сообщения, но не удаляет историю автоматически. Архивация или удаление персонального агента остаётся отдельной административной операцией с резервной копией.
- При выключении функции существующие точные binding продолжают работать; массовое удаление или объединение памяти не выполняется.

## Проверка

Реализация выполняется через тесты. Обязательные сценарии:

1. функция и config writes выключены;
2. групповое сообщение;
3. policy не `open`;
4. уже существующая точная привязка;
5. ручной маршрут не в `sourceAgentId`;
6. успешное создание агента и binding;
7. первое сообщение после создания маршрутизируется новому агенту;
8. повторное и одновременное первое сообщение не создают дубликаты;
9. runtime-конфигурация новее входного снимка;
10. достижение per-account `maxAgents` и отказ от shared fallback;
11. одинаковый `userId` в разных аккаунтах создаёт разные агенты;
12. существующий агент без binding получает только недостающую привязку;
13. запрещённый путь bootstrap и отсутствие копирования памяти/результатов;
14. direct routing использует `fromUserId`, а доставка — `dialogId`;
15. обе копии manifest schema совпадают;
16. расчёт НДС 22/122 на граничных суммах и одинаковое округление всех итогов.

Локальная приёмка включает `npx vitest run`, `npm run lint`, сборку плагина и проверку отсутствия `openclaw/*` imports.

Серверная приёмка включает резервную копию, проверку конфигурации OpenClaw, включение функции на всех четырёх аккаунтах, перезапуск/перезагрузку без ошибок и реальные сообщения тестового сотрудника каждому Bitrix24-боту. Для ТКП дополнительно проверяются сгенерированный DOCX/PDF, сумма НДС 22%, сохранность корпоративного бланка и фактическая доставка файла в Bitrix24.

## Границы этой доработки

Входит в работу:

- общий механизм динамических агентов в Bitrix24-плагине;
- конфигурация всех четырёх клиентских ботов;
- персональные workspace и memory;
- безопасный bootstrap из соответствующего базового агента;
- исправление и тестирование НДС 22% в генераторе ТКП;
- запрет обхода корпоративного генератора и шаблона;
- локальная и серверная проверка с реальным Bitrix24-сообщением.

Не входит в эту доработку:

- отдельный видимый бот для каждого сотрудника;
- общая память одного сотрудника между разными функциональными ботами;
- автоматическое удаление данных уволенного сотрудника;
- веб-интерфейс управления персональными агентами;
- редактирование корпоративного бланка сотрудниками.
